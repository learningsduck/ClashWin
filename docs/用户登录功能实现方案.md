# Clash Verge Rev - 用户登录功能实现方案

## 一、项目结构简要分析

本项目是 **Clash Verge Rev**，技术栈如下：

| 层级 | 技术 | 关键路径 |
|------|------|----------|
| 前端 | React 19 + React Router 7 + MUI 7 + Vite 7 | `src/` |
| 后端 | Tauri 2 (Rust) | `src-tauri/` |
| 路由 | 单层 Layout，所有页面为子路由 | `src/pages/_routers.tsx`、`_layout.tsx` |
| 配置 | Verge 配置（Rust 读写） | `get_verge_config` / `patch_verge_config` |
| 命令 | 通过 `invoke("cmd_name", args)` 调用 Rust | `src-tauri/src/cmd/mod.rs`、`lib.rs` |

当前**没有**任何用户/登录体系，应用启动后直接进入主界面（首页、代理、配置等）。

---

## 二、需求与对应方案

| 需求 | 实现思路 |
|------|----------|
| 用户登录界面 | 新增 `/login` 页面，未登录时强制跳转至此 |
| 手机号 + 短信验证登录 | 前端调你的**认证后端**：发验证码 → 提交手机号+验证码 → 换取 token |
| 自动登录 | 本地安全存储 token，启动时带 token 请求后端校验，通过则直接进主界面 |
| 换设备后旧设备自动断线 | 后端「单设备」：新设备登录时使旧 session 失效，旧端请求返回 401/会话失效，前端清 token 并跳转登录 |

短信发送、会话管理、踢线逻辑都必须在**你自己部署的后端**完成，客户端只负责：调接口、存 token、做登录态校验和掉线处理。

---

## 三、整体架构

```
┌─────────────────┐     HTTPS      ┌──────────────────┐
│  Clash Verge    │ ◄─────────────►│  你的认证后端     │
│  (Tauri 客户端)  │  登录/校验/踢线  │  (自建或 BaaS)   │
└────────┬────────┘                └────────┬─────────┘
         │                                  │
         │ 本地存储 token                    │ 短信服务
         ▼                                  ▼
┌─────────────────┐                ┌──────────────────┐
│  token/refresh  │                │  阿里云/腾讯云等   │
│  (安全存储)      │                │  短信 API         │
└─────────────────┘                └──────────────────┘
```

## 四、实施步骤（推荐顺序）

### 阶段一：认证后端（必须先有）

1. **选型**
   - **自建**：Node.js / Go / Python 等，提供 REST API。
   - **或** 使用 BaaS（如 Firebase Auth、Auth0），再配合自建一个小 API 做「单设备踢线」和业务逻辑。

2. **后端必备接口示例**

   | 接口 | 方法 | 说明 |
   |------|------|------|
   | `/auth/sms/send` | POST |  body: `{ "phone": "+86xxx" }`，发短信验证码，限频、防刷 |
   | `/auth/sms/login` | POST |  body: `{ "phone": "+86xxx", "code": "123456", "device_id": "xxx" }`，校验通过返回 `access_token`、可选 `refresh_token` |
   | `/auth/me` 或 `/auth/validate` | GET |  Header: `Authorization: Bearer <token>`，校验 token 是否有效、是否被踢 |
   | `/auth/refresh` | POST |  用 refresh_token 换新 token（可选） |
   | `/auth/logout` | POST |  当前 token 失效（可选） |

3. **单设备 / 踢线逻辑（后端）**
   - 用户表：`user_id`、`phone` 等。
   - 会话表：`user_id`、`token`/`session_id`、`device_id`、`created_at`。
   - 策略：**同一用户只允许一个有效会话**。  
     - 在 `POST /auth/sms/login` 时：  
       - 为该用户生成新 session，写入新 token、device_id。  
       - 将该用户其它 session 标记为失效（或删除）。  
     - 在 `GET /auth/me` 或业务接口中：若带上的 token 对应 session 已被标记失效，返回 **401** 且 body 可带 `code: "SESSION_REPLACED"`（或类似），前端据此提示「您的账号已在其他设备登录」并清空本地、跳转登录。

4. **短信**
   - 国内：阿里云、腾讯云等「短信验证码」产品，在后端调用其 API 发验证码。
   - 后端需做：验证码生成、过期时间、同一手机号限频、防刷。

---

### 阶段二：前端 - 登录页与路由守卫

1. **新增登录页**
   - 在 `src/pages/` 下新建 `login.tsx`。
   - 内容建议：
     - 手机号输入（带国家区号可选）。
     - 「获取验证码」按钮（调后端 `POST /auth/sms/send`，成功后倒计时）。
     - 验证码输入框。
     - 「登录」按钮：调 `POST /auth/sms/login`，成功后拿到 token。

2. **路由与守卫**
   - 在 `src/pages/_routers.tsx` 中：
     - 增加一条**不在 Layout 下**的独立路由：`path: "/login"`, `Component: LoginPage`。
     - 主 Layout 的 `path: "/"` 下保持现有首页、代理、配置等子路由。
   - 实现「未登录只能看 /login」：
     - **方案 A**：在 `_layout.tsx` 或包裹 Layout 的父级里，根据「是否已登录」决定渲染 `<Outlet />` 还是 `<Navigate to="/login" />`。  
     - **方案 B**：使用 React Router 的 loader/guard：在根或 Layout 的 loader 里检查登录态，未登录则 `redirect("/login")`。  
   - 登录成功后：`navigate("/")` 或 `navigate("/home")` 进入主界面。

3. **登录态来源**
   - 用 React Context 或全局 state（如 Zustand）保存：`token`、可选 `userInfo`。
   - 初始值：从「本地存储」读（见下）。

---

### 阶段三：Token 存储与自动登录

1. **存储位置（二选一或组合）**
   - **仅前端**：  
     - 用 Tauri 的 `@tauri-apps/plugin-store` 或写入本地文件（通过 Rust 命令读写），避免明文放在 localStorage。  
     - 或先用 localStorage 做最小实现，后续再迁到 Tauri store/文件。
   - **Rust 侧**：  
     - 在 `src-tauri/src/cmd/` 下新增 `auth.rs`，提供：  
       - `get_auth_token() -> Option<String>`  
       - `set_auth_token(token: String)`  
       - `clear_auth_token()`  
     - 实现上可写到一个单独配置文件（如 `auth.json` 或与 verge 配置同目录下的 `auth_store`），注意不要提交到版本库。

2. **自动登录流程**
   - 应用启动（如 `main.tsx` 或 Layout 首次挂载）：
     1. 读取本地 token（上述存储）。
     2. 若无 token → 视为未登录，展示/跳转 `/login`。
     3. 若有 token → 请求后端 `GET /auth/me`（或 `/auth/validate`）。
        - 200 → 认为已登录，写入 Context/state，渲染主界面。
        - 401 且 `code === "SESSION_REPLACED"` → 清除本地 token，提示「您的账号已在其他设备登录」，再跳转 `/login`。
        - 其他 401/5xx → 清除 token（可选），跳转 `/login`。

---

### 阶段四：换设备后旧设备自动断线（前端配合）

1. **后端**（阶段一已做）
   - 新设备登录时，使该用户旧 session 失效。
   - 旧设备后续任何带 token 的请求返回 401 + `SESSION_REPLACED`。

2. **前端**
   - 对所有请求（如 axios 的 response interceptor）做统一处理：
     - 若 status === 401 且 body.code === `"SESSION_REPLACED"`：  
       - 清除本地 token、清空登录 state。  
       - 跳转 `/login`，并提示「您的账号已在其他设备登录，请重新登录」。
   - 可选：定时轮询 `GET /auth/me` 或由后端提供 WebSocket/SSE，实现「实时踢线」，体验更好。

## 六、注意事项

1. **HTTPS**：生产环境认证接口必须使用 HTTPS。
2. **Token 安全**：尽量用 Tauri 侧或加密存储，避免明文长期放在 localStorage。
3. **设备 ID**：可用 `@tauri-apps/api` 的 `os.platform()`、机器名等生成稳定 device_id，用于后端区分设备并在踢线时提示「某设备」。
4. **合规**：手机号收集与短信验证需符合隐私政策与当地法规。

按「阶段一 → 阶段二 → 阶段三 → 阶段四」顺序做，即可实现：登录页、短信验证、自动登录、换设备后旧设备自动断线。若你希望，我可以按上述清单从「登录页 + 路由守卫」或「Rust auth 命令」开始，给出对应文件的具体代码示例（需你提供后端 base URL 和接口约定）。
